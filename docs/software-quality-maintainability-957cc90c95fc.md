# 软件可维护性

> 原文：<https://medium.datadriveninvestor.com/software-quality-maintainability-957cc90c95fc?source=collection_archive---------31----------------------->

![](img/e13c5ec00bc14ab38f697e071b09f51f.png)

软件可维护性是对问题的理解程度，以及为增强和简化可读性、可理解性、可靠性和安全性以及适应性而构建的解决方案。

这不仅仅是构建一个解决方案来解决某个问题。它深入到诸如代码结构、认知复杂性和程序质量等你能想到的细节。

正如我所意识到的，软件可维护性比编程的其他阶段需要更多的努力。然而，作为一名程序员/开发人员，很容易在编写代码时陷入困境，忘记约定，或者忘记您正在构建的系统是供他人而不是自己使用的，或者忘记这个系统可能在某个时候需要进行维护。我们很多人都是这种情况的受害者，我也是。

像 [code climate](https://codeclimate.com/) 这样的开源平台有助于自动化确保代码健康的过程。我最近使用这个平台来分析我的程序的可维护性，并被一个欢呼 D 震惊了！供你参考，这是一个有很多代码味道的 20 %- 50 %:重复，每个函数太多参数，太多返回，太多嵌套循环，高[认知复杂性](https://docs.codeclimate.com/docs/cognitive-complexity)等等。我的意思是，这个程序正在做它应该做的事情，但是如果我两年后回到这个程序，我打赌我是否能够理解一件事情。

克服这一挑战与重构代码息息相关。这是一个有益的过程，但是可以通过正确理解来自测试平台的反馈、理解潜在的概念以及回顾如何最大化这些信息来产生高质量的代码来完成。

我认为得到 D 并不是可能发生的最糟糕的情况，但我开始考虑如何最好地组织我的代码，以一直达到 a。在我的情况下，代码环境提供了一些指示，因此帮助我识别了我的所有问题。我的工作是通过以下方式扭转这些问题:在一个函数中最多有 4 个参数，最小化嵌套循环，仅在必要时使用返回，等等。
